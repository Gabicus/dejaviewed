Thesis

Autonomous agents went from demo-ware to daily drivers in 2026. But the gap between 'impressive demo' and 'reliable tool' is massive. These entries map the frontier: Qwen agents roaming NYC streets, Claude automating daily workflows, Kronos failing hedge fund stress tests. The pattern: narrow agents with clear guardrails work; general agents with vague goals don't.

Why It Matters

Everyone's building agents. Most are failing. The entries in this dive separate signal from noise: what agent patterns actually work in production, what fails under stress, and where the reliability boundary sits. The common thread: agents succeed when scope is narrow, feedback is fast, and failure is cheap.

Action Sketch

  1. Start narrow: one task, one tool, one feedback loop
  2. Build in guardrails before capability (kill switches, budget limits, scope constraints)
  3. Test adversarially: what happens when the agent encounters edge cases?
  4. Add persistence: agents need memory to avoid repeating mistakes
  5. Monitor: log everything, review regularly, prune bad patterns
  6. Scale horizontally: multiple narrow agents > one general agent

Prerequisites

  • Understanding of LLM capabilities and limits
  • Experience with at least one agent framework
  • Tolerance for debugging non-deterministic systems
